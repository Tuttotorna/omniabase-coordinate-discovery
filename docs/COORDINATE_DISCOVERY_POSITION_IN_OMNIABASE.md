# OMNIABASE COORDINATE DISCOVERY — POSITION IN OMNIABASE

## Status

This document defines the position of **Omniabase Coordinate Discovery** inside the broader **OMNIABASE** framework.

Its purpose is to prevent architectural confusion.

This repository is not the whole framework.
It is one of its canonical branches.

This document freezes that distinction.

---

## 1. General framework vs branch

**OMNIABASE** is the general framework.

It defines the broad multirepresentational principle:

> a phenomenon should not be treated as exhausted by a single representation.

From that principle, multiple operational directions follow.

**Omniabase Coordinate Discovery** is one of those directions.

More precisely:

**Omniabase Coordinate Discovery is the Coordinate Discovery branch of OMNIABASE.**

This means:

- OMNIABASE is broader than this repository
- this repository inherits its foundation from OMNIABASE
- this repository does not define the full scope of OMNIABASE
- this repository is one operational consequence of the Omniabase principle

---

## 2. What OMNIABASE asks

At the most general level, OMNIABASE asks:

- what depends on the representation?
- what remains stable beyond it?
- what emerges only through recoding?
- what collapses when representational privilege is removed?
- what structure becomes visible only beyond a single view?

These questions define the framework.

They are broader than any one implementation.

---

## 3. What Coordinate Discovery asks

This branch asks a narrower and more specific question:

> what structure becomes visible only when a phenomenon is observed across multiple codings rather than a single one?

This is the coordinate-discovery question.

It focuses on the possibility that representational variation may expose:

- hidden coordinates
- latent factors
- regime separations
- generator distinctions
- invisible symmetries
- more faithful descriptive axes

This is its proper scope.

---

## 4. Coordinate Discovery as canonical branch

Inside the canonical Omniabase taxonomy, this repository belongs to the branch called:

**Omniabase Coordinate Discovery**

This branch is defined by one core function:

to use multiple representations in order to expose structure that standard single-view descriptions may compress, blur, or hide.

Its typical outputs include:

- new descriptive coordinates
- latent-variable candidates
- structural maps
- regime separation signals
- generator-type distinctions
- coordinate suggestions for modeling and further study

This branch is experimental by design.
Its value lies in exposing hidden structural possibilities, not in claiming premature closure.

---

## 5. What this branch is not inside the framework

Inside OMNIABASE, this repository should not be confused with:

- the umbrella framework itself
- the Diagnostics / Structural Measurement branch
- the Cross-Representation Translation branch
- the epistemic pre-layer associated with observer decentering
- the total history of related MB-X.01 repositories

This repository is not the general theory of the ecosystem.

It is one branch with one architectural role.

That role is generative, exploratory, and structurally disciplined.

---

## 6. Canonical neighboring branches

To understand this repository correctly, it should be read together with the existence of the other canonical Omniabase branches.

### A. Diagnostics

Question:

> when something looks stable in one representation, does that stability survive when representation changes?

This branch is more defensive than Coordinate Discovery.

It is concerned with:

- robustness
- fragility
- drift
- saturation
- collapse proximity
- representation dependence

That is the role of **OMNIA** and related repositories.

This repository does not primarily do that.

---

### B. Cross-Representation Translation

Question:

> when two descriptions appear different, how much are they still describing the same structural object?

This branch is more interface-oriented than Coordinate Discovery.

It is concerned with:

- compatibility
- alignment
- translatability
- invariant residue across descriptions
- structural bridges across formalisms

This repository does not primarily do that either.

---

## 7. Relationship to observer decentering

The Omniabase ecosystem also includes an epistemic pre-layer concerned with reducing the silent monopoly of the privileged observer.

That work helps motivate why a single representation should not be treated as final.

Coordinate Discovery inherits the consequences of that move, but it is not itself the full observer-decentering framework.

It assumes the need for representational variation.
It does not exist to establish that need from scratch.

---

## 8. Architectural rule

The following distinction must remain stable:

- **OMNIABASE** -> general framework
- **OMNIA** -> Diagnostics / Structural Measurement branch
- **omniabase-coordinate-discovery** -> Coordinate Discovery branch
- **translation-related repositories** -> Cross-Representation Translation branch
- **pre-layers and satellites** -> conceptually relevant, but not identical to the framework core

This distinction is not cosmetic.

It protects both:

- the breadth of OMNIABASE
- the precision of this branch

Without it, the ecosystem collapses into conceptual ambiguity.

---

## 9. Why this distinction matters

If Coordinate Discovery is treated as the whole of OMNIABASE, two errors follow.

### Error 1

The framework becomes artificially narrowed.

OMNIABASE would then be mistaken for:

- only multibase signal probing
- only latent-coordinate search
- only dynamical-system exploration
- only one style of structural discovery

That would be false.

### Error 2

This repository itself becomes overloaded.

It would then be expected to cover:

- diagnostics and fragility sensing
- translation across representations
- all epistemic functions
- the full philosophical foundation

That would also be false.

The distinction therefore protects conceptual accuracy.

---

## 10. This branch's proper strength

Omniabase Coordinate Discovery is strongest when described exactly for what it is:

an experimental multibase structural discovery framework that tests whether hidden structure becomes visible under representational variation.

That is enough.

It does not need to become larger by absorbing the whole framework.

Its value comes from disciplined generativity, not inflation.

---

## 11. Strategic role

Inside the Omniabase ecosystem, this branch serves a strategic role:

it is one of the clearest expressions of the more exploratory and frontier-oriented side of OMNIABASE.

Why?

Because it aims not only to test whether structure holds, but to explore whether representational variation can expose:

- new coordinates
- latent separations
- previously compressed distinctions
- structurally useful descriptive axes

That role makes this branch especially important for the long-term identity of the framework.

That strategic importance should not be confused with architectural centrality.

The framework is broader than this branch.
This branch is one way the framework opens access to hidden structure.

---

## 12. Summary formula

The shortest correct formula is this:

**OMNIABASE is the general multirepresentational framework.  
Omniabase Coordinate Discovery is its branch for exposing hidden coordinates, latent structure, and regime separations through representational variation.**

Or more explicitly:

**OMNIABASE defines the principle.  
Omniabase Coordinate Discovery applies that principle to the problem of revealing structure that standard single-view representations may fail to expose.**

That is the correct position of this repository inside OMNIABASE.
